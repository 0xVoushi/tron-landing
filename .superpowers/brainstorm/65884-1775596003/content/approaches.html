Three Approaches for the Hero Section Architecture

All use Next.js 15 + Tailwind CSS + TypeScript. Differences are in component structure and CSS strategy.

A

Single-File Component Recommended

One HeroSection.tsx with all sub-elements inline. Clip-path and glassmorphism live in globals.css as utility classes. Tailwind handles everything else.

Pros

  • Fast to build
  • Easy to hand off
  • Low overhead for a single page

Cons

  • File grows if more sections are added later
B

Decomposed Components

Split into Navbar, VideoBackground, HeroContent, ConsultationCard, and a shared ClippedButton — each in its own file under components/.

Pros

  • Clean separation of concerns
  • ClippedButton reusable across future sections
  • Easier to test individually

Cons

  • More files for the same output
  • Slight over-engineering for a hero-only build
C

CSS Modules for Specialised Effects

Tailwind for layout/spacing, HeroSection.module.css for the clip-path and glassmorphism. Keeps Tailwind classes clean and avoids arbitrary-value verbosity.

Pros

  • Custom effects stay scoped and readable
  • No Tailwind [] hack for complex clip-path

Cons

  • Mixes two styling systems
  • Extra context-switching during development

My recommendation

Option A for now. The clip-path and glassmorphism are just two CSS utility classes — not enough complexity to justify a separate module. If more sections are added later, we can split components out without rewriting anything.
