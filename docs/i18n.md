# Internationalization (i18n)

`tron-landing` ships with `next-intl` v4 wired into the Next.js App Router.
Every page lives under `src/app/[locale]/...` and copy is read from
`messages/<locale>.json`.

## Locales

The single source of truth is [src/i18n/locales.ts](../src/i18n/locales.ts):

| Code | Display name | hreflang `iso` | Default? |
|------|--------------|----------------|----------|
| en   | English      | en             | ✅       |
| ru   | Русский      | ru             |          |
| zh   | 中文          | zh-Hans        |          |
| ko   | 한국어         | ko             |          |
| pt   | Português    | pt             |          |
| es   | Español      | es             |          |
| fr   | Français     | fr             |          |
| de   | Deutsch      | de             |          |

Routing uses `localePrefix: 'as-needed'` — the default (`en`) is unprefixed,
all others live under `/<code>/...`.

## Add a new language

1. Append the new entry to [src/i18n/locales.ts](../src/i18n/locales.ts).
2. Copy `messages/en.json` to `messages/<new-code>.json`. Keep the
   `"_translationStatus": "pending"` marker until the file is fully translated.
3. Run `npm run i18n:check` — it must exit clean (key parity with `en`).

The middleware, sitemap, hreflang `<link>` entries, locale switcher, and
`generateStaticParams` all read from the registry, so no other file needs to
change.

## Authoring keys

- Namespaces mirror the information architecture: `nav.*`, `hero.*`, `faq.items.<key>.*`,
  `footer.links.*`, etc.
- Use **named** placeholders only: `"© {startYear}–{currentYear} TRON Multisender"`.
  Never use positional `{0}`-style.
- Keep proper nouns out of JSON when they are stable across languages
  (`TRON`, `USDT`, `TRC-20`, `TronLink`). Inline them in the source string.
- For ordered structural lists (steps, FAQ, plans, benefits), the order is
  encoded in TypeScript as a `*_KEYS` const tuple
  (e.g. [src/data/faq.ts](../src/data/faq.ts), [src/data/guide.ts](../src/data/guide.ts)).
  Translators only edit the JSON values; ordering is locked in code.

## Metadata, hreflang & structured data

- `<title>` / `<meta description>` come from
  [src/lib/metadata.ts](../src/lib/metadata.ts) via `buildMetadata({ locale, titleKey, descriptionKey, path })`.
- `alternates.canonical` points at the locale-correct URL; `alternates.languages`
  contains an entry for every locale (`iso` field) plus an `x-default` pointing
  at the default-locale URL.
- Per-page `generateMetadata` calls only pass key paths — never raw strings.
- JSON-LD lives in [src/lib/structured-data.ts](../src/lib/structured-data.ts);
  `Organization`, `SoftwareApplication`, `HowTo`, `WebSite`, and `FAQPage`
  schemas are rebuilt per locale with translated text.

## Locale switcher

[`LocaleSwitcher`](../src/components/locale-switcher/LocaleSwitcher.tsx) is a
client component used inside the Navbar (desktop right-side; mobile menu).
It reads the current locale + pathname from `next-intl` and renders the
remaining locales as locale-prefixed `Link`s. The cookie name is `ms_locale`,
matching the Multisender Nuxt landing's `ms_i18n` convention.

## Internal links

ESLint blocks `next/link` and the locale-unaware navigation helpers from
`next/navigation`. Always import from `@/i18n/routing`:

```tsx
import { Link, usePathname, useRouter, redirect } from '@/i18n/routing'
```

Exceptions:
- [src/components/footer/SocialLinks.tsx](../src/components/footer/SocialLinks.tsx)
  links to external destinations and is exempt from the rule.

## Formatters

Number, currency, percent, and date formats are declared in
[src/i18n/formats.ts](../src/i18n/formats.ts). Use them via
`useFormatter()`/`getFormatter()` from `next-intl`. They are wired into both
the request config and the `NextIntlClientProvider`.

## CI: key parity

`npm run i18n:check` walks every `messages/*.json` and diffs key paths
against `en.json`. Missing keys fail the build; extra keys warn. The script
also reports each file's `_translationStatus` value.

## Tests

`src/test/render.tsx` exports `renderWithIntl(ui, { locale?, messages? })`
which wraps the tree in `NextIntlClientProvider` with the English dictionary
by default. All component tests use this helper instead of plain `render`.

## Known gaps

- **CJK glyphs**: Rubik has no CJK coverage; `zh` and `ko` rely on the
  system font fallback. Acceptable for a brand display font.
- **Per-locale OG images**: `buildMetadata` accepts an `ogImage` override but
  no localized art has been authored yet — all locales currently share
  `/og-image.png`.
- **Translations**: non-English dictionaries are scaffolded as copies of
  `en.json` with `"_translationStatus": "pending"`. Translation is a
  follow-up task.
